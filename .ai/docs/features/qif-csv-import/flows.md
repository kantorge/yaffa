# QIF/CSV Import — Flows

Only flows that touch permissions, data integrity, external calls, or privacy are included, per the anti-PRD rule. Each flow lists actor, precondition, success outcome, the step sequence, the authz check at each protected step (with the expected deny case), and trust-boundary crossings / side effects.

## a) Parse-upload flow — CSV

**Actor**: authenticated, verified user. **Precondition**: user owns an `account_entities` row with `config_type = 'account'`; a CSV `FileImportProfile` is either explicitly selected or set as that account's `preferred_file_import_profile_id`. **Success outcome**: a JSON list of draft transactions is returned to the browser; nothing is persisted.

1. Browser POSTs multipart `file`, `source_type=csv`, `account_id`, optional `file_import_profile_id` to `POST /api/v1/imports/parse` (`ImportUploadCard.vue` → `ImportPage.vue:onSubmit`, `ImportApiController::parse`, `app/Http/Controllers/API/ImportApiController.php:38`). **Trust boundary**: browser → server file upload.
2. `ImportParseRequest` validates: `source_type` in `[qif, csv]`, `account_id` exists, `file` is `qif|txt|csv` and ≤ `config('yaffa.import_max_file_size_mb')` MB, and — if `file_import_profile_id` given — that it resolves via `FileImportProfile::selectableForUser($user)` and matches the `source_type` (`ImportParseRequest.php:40-69`).
3. **Authz check 1**: `Gate::authorize('import.parse', $accountEntity)` → `ImportPolicy::parse` requires `$user->id === $accountEntity->user_id && $accountEntity->isAccount()` (`ImportPolicy.php:10`). **Deny case**: importing into another user's account, or into a non-account `AccountEntity` (e.g. a payee row) → 403.
4. Profile resolution (`resolveFileImportProfile`, `ImportApiController.php:135`): if `file_import_profile_id` given, re-fetches it unscoped and re-runs **authz check 2**: `Gate::authorize('view', $profile)` → `FileImportProfilePolicy::view` = `isSystem() || isUserOwnedBy($user)` (`FileImportProfilePolicy.php:15`). **Deny case**: selecting another user's `type=user` profile → 403 even though step 2's form-request check would already have caught it — this is defense in depth, not the only gate. If no ID given, falls back to `$accountEntity->preferred_file_import_profile_id` and re-checks the same `view` gate on that FK target (in case the FK was set before a profile's ownership changed, though there's no code path that changes ownership post-creation).
5. If no profile resolves at all → `422 CSV_PROFILE_REQUIRED`, no further processing.
6. `CsvParserService::parseFile()` reads the file server-side, normalizes encoding, applies the account's/system's mapping and (for system profiles) `matching_rules`/`actions`, enforcing `import_max_rows`; throws `RuntimeException` → `422 IMPORT_PARSE_FAILED` if the cap is exceeded or the CSV is unparseable.
7. `ImportNormalizationService::enrichDraftsWithPayeeMatches($user, ...)` — reads only `$user->payees()` (`ImportNormalizationService.php:94`, via `$user->payees()` relation — owner-scoped).
8. `ImportDuplicateDetectionService::enrichDrafts($user, ...)` — reads only `$user->transactions()` within the combined draft date window (`ImportDuplicateDetectionService.php:119`).
9. `ImportNormalizationService::enrichDraftsWithRelatedAiDocuments($user, ...)` — reads only `AiDocument::where('user_id', $user->id)` (`ImportNormalizationService.php:305`).
10. Response returned to browser. **Side effects**: none (no DB writes). **Trust boundary crossings**: browser→server file read only; no outbound calls.

## b) Parse-upload flow — QIF

Same entry point and same account-ownership check (step 3 above) as the CSV flow. Differences:

1. `source_type=qif`; profile is optional (QIF works without one). If `file_import_profile_id` given, `resolveQifImportProfile` (`ImportApiController.php:114`) re-runs the same `Gate::authorize('view', $profile)` check and additionally requires `$profile->file_type === 'qif'`.
2. `QifParserService::parseFile()` reads the file, line-parses `!Type:Bank/Cash/CCard` sections (other section types are skipped with a warning, not rejected), applies the profile's `field_map`/`amount_sign` overrides if a profile was supplied, and enforces `import_max_rows` on entry count (`QifParserService.php:151,221`).
3. `ImportNormalizationService::normalizeQifEntries()` converts raw QIF fields to the same draft shape as the CSV path (no `AccountEntity`/user lookups at this stage — normalization is pure).
4. Same three enrichment steps as the CSV flow (payee match, duplicate detection, related AI documents) run identically afterward.

## c) User profile CRUD (create / update / delete + affected-accounts warning)

**Actor**: authenticated user. **Precondition** (update/delete): profile exists. **Success outcome**: profile row created/mutated/removed, scoped to the owner.

- **Create** — `POST /api/v1/imports/file-profiles` (`FileImportProfileApiController::store`, line 56). **Authz**: `Gate::authorize('create', FileImportProfile::class)` → policy always allows (`FileImportProfilePolicy::create` returns `true` unconditionally, line 20 — anyone verified can create). `FileImportProfileRequest` marks `user_id`, `key`, `type` as `'prohibited'` in the request body (`FileImportProfileRequest.php:36-38`); the controller then hard-sets `$profile->user_id = $user?->id; $profile->type = 'user';` (`FileImportProfileApiController.php:75-76`) — a client cannot mint a system profile or assign a profile to another user via this endpoint, this is enforced in code, not just validation.
- **Update** — `PATCH /api/v1/imports/file-profiles/{profile}`. **Authz**: `Gate::authorize('update', $profile)` → `FileImportProfilePolicy::update` = `isUserOwnedBy($user)` (line 25). **Deny case**: a system profile (`user_id` is always `null` for `type=system`) can never satisfy `isUserOwnedBy`, so **no user can ever PATCH a system profile** — enforced server-side by the policy, not merely hidden in the UI. `file_type` is `prohibited` on update (immutable after creation).
- **Delete** — `DELETE /api/v1/imports/file-profiles/{profile}`. **Authz**: `Gate::authorize('delete', $profile)` → same `isUserOwnedBy` check, same system-profile deny. **Data-integrity check**: before deleting, the controller checks `$profile->accountEntities()->exists()` and returns `422` if any account still references this profile as its `preferred_file_import_profile_id` (`FileImportProfileApiController.php:96-100`) — this is a server-side guard, not just a client warning.
- **Affected-accounts warning (client)**: `FileImportProfileManager.vue` fetches `account_entities` alongside each profile (via `index`'s eager load scoped to `$user->id`, line 38-41) and **disables the delete button entirely** (with a tooltip listing affected account names) whenever `profile.account_entities.length > 0` (`FileImportProfileManager.vue:433-449`). A separate `deleteProfile()` method (line 713) builds a Swal confirmation dialog that *also* includes affected-account text, but given the button is disabled first, this code path is effectively unreachable in the current UI — it reads as leftover/defensive code from an earlier iteration where delete-with-warning was allowed. **Drift from spec**: `SPECIFICATION.md` describes a dedicated `GET /imports/file-profiles/{profile}/affected-accounts` endpoint (§ "Endpoints (draft)"); the shipped code has no such route — the same data is piggybacked onto the `index` response's `with(['accountEntities' => ...])` eager load instead.

## d) AI-assisted profile suggestion

**Actor**: authenticated user with a configured `AiProviderConfig`. **Precondition**: user has saved their own AI provider config (BYO key). **Success outcome**: a structured profile suggestion (delimiter, date format, column mapping, etc.) is returned for form pre-fill; nothing is persisted.

1. Browser POSTs multipart `file` (+ optional `account_id`) to `POST /api/v1/imports/file-profiles/suggest` (`FileImportProfileApiController::suggest`, line 107). **Trust boundary**: browser → server file upload.
2. `SuggestFileImportProfileRequest` validates `file` is `csv|txt` and ≤ `import_max_file_size_mb` × 1024 KB; `account_id` optional, must exist.
3. Controller looks up `AiProviderConfig::where('user_id', $user?->id)->first()` — **scoped to the caller**, never another user's config. If none exists → `422` with a message directing the user to configure a provider first (no fallback to a shared/system key).
4. **Authz check** (only if `account_id` given): `Gate::authorize('view', $account)` on the `AccountEntity` (line 131) — reuses the account policy; the account is only used for prompt context (its name), it is not written to.
5. `AiImportProfileSuggestionService::suggest()` trims the CSV to header + first 10 data rows (`MAX_SAMPLE_DATA_ROWS`, line 22) and builds a prompt.
6. **Trust boundary crossing — server → external AI provider**: `Prism::structured()->using($config->provider, $config->model)->usingProviderConfig(['api_key' => $config->api_key])->...->asStructured()` (line 118) sends the trimmed CSV sample (real transaction dates/amounts/payee/merchant text) to whichever provider the user configured, authenticated with that user's own decrypted API key. Request timeout is `PRISM_REQUEST_TIMEOUT` (global Prism config, not import-specific despite the `.env.example` comment above it — see `variables.md`).
7. Response is schema-validated by Prism (`ObjectSchema`) and then `sanitizeResponse()` strips any AI-suggested `field` name that isn't a known `ImportCanonicalField`, converting it to a `confidence_notes` entry instead of silently accepting arbitrary field names (line 172). **Side effect**: none persisted; the client uses the response to pre-fill the profile creation wizard, and the user still must submit `POST /imports/file-profiles` (going through the same `create` authz + `FileImportProfileRequest` validation as any manual profile) to actually save it.
8. Provider-side failures are wrapped in `AiProviderFailureException` → `502`; parse/read failures → `422`.

## e) Draft finalization via the existing transaction endpoint

**Actor**: authenticated user reviewing drafts from flow (a) or (b). **Precondition**: at least one `pending_review` draft exists in the browser's in-memory state (never server-side). **Success outcome**: exactly one `Transaction` row created, same as manual entry.

1. User clicks "finalize" on a draft row → `ImportDraftTable` → `ImportPage.vue:onFinalizeDraft` (line 415) builds a transaction-shaped object from the draft (`transaction_type`, `date`, `config.account_from_id/account_to_id`, `config.amount_from/amount_to`) and dispatches `window.dispatchEvent(new CustomEvent('initiateCreateFromDraft', ...))`.
2. This event is consumed by the **pre-existing** transaction-create Vue island (shared with AI document processing and manual entry — not part of this feature's diff), which opens its create modal pre-filled and, on submit, POSTs to `POST /api/v1/transactions/store-standard` — the same endpoint, same `TransactionRequest` validation, same `Gate`/ownership checks as manual transaction creation (out of scope of this feature's code; see the transaction feature's own docs for that endpoint's authz).
3. On success, the transaction-create island fires `window.dispatchEvent(new CustomEvent('transaction-created'))`; `ImportPage.vue:onTransactionCreated` (line 457) marks the corresponding draft `status: 'finalized'` in local state only.
4. **No import-specific write happens anywhere in this flow.** There is no linkage recorded between the finalized transaction and its source draft/file (matches `SPECIFICATION.md`'s "Storage and Retention" section, which explicitly says this is acceptable because sessions are one-off). **Side effect**: one `Transaction` (+ items) row via the existing endpoint's own logic.
5. **Idempotency**: no server-side guard against a rapid double-click creating two transactions from the same draft; the spec explicitly delegates this to the frontend disabling the finalize control after first click (`SPECIFICATION.md` § Concurrency and Idempotency), and the shipped `ImportPage.vue` tracks `finalizingDraftIndex` but does not visibly disable the button while a create request from the *other* island is in flight — this coupling across two independently-written Vue islands is worth a closer look before shipping (flagged, not confirmed as broken, since the create-modal itself may have its own submit-guard that is out of scope here).

## f) System profile sync on deploy

**Actor**: none (deploy-time automation, runs as the container's entrypoint user). **Precondition**: container starts. **Success outcome**: `file_import_profiles` rows with `type='system'` reflect exactly what's hardcoded in `SystemFileImportProfileRegistry::profiles()`.

1. `docker/entrypoint.sh:17` runs `php artisan app:import:sync-system-profiles --no-interaction` unconditionally on every container start (no flag to skip it).
2. `SyncSystemFileImportProfilesCommand::handle()` iterates the registry's hardcoded array (currently 2 profiles: `hun_raiffeisen_v1` CSV, `qif_swap_p_m_v1` QIF) and does `FileImportProfile::firstOrNew(['key' => ...])->fill(...)->save()`, force-setting `user_id = null` and `type = 'system'` regardless of prior DB state (`SyncSystemFileImportProfilesCommand.php:20-37`).
3. **No authz check applies** — this is a trusted, code-only input path (no user input reaches the registry). **Trust boundary**: none crossed at runtime; the trust boundary is at code-review/deploy time (whoever can merge to the registry file controls what every user sees as a "system" profile, including its `matching_rules` regexes — see `architecture.md`'s ReDoS note).
4. **Side effect**: upsert only; existing `type=system` rows not present in the registry are never deleted by this command (no orphan cleanup), and any manual DB edit to a system profile row is silently overwritten on next deploy.
