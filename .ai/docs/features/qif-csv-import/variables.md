# QIF/CSV Import — Variables

Only variables actually referenced by import-feature code are listed (grepped for `env(`/`config(` under `app/Services/Import/`, `app/Http/Controllers/API/ImportApiController.php`, `app/Http/Controllers/API/FileImportProfileApiController.php`, `app/Http/Requests/ImportParseRequest.php`, `app/Http/Requests/SuggestFileImportProfileRequest.php`, `app/Console/Commands/SyncSystemFileImportProfilesCommand.php`). None are read directly with `env()` inside these files — all go through `config('yaffa.*')`, which itself reads `env()` in `config/yaffa.php`.

## Env Vars Actually Used

| Name | Used by | Scope | Source | Risk |
|---|---|---|---|---|
| `IMPORT_MAX_FILE_SIZE_MB` | `config('yaffa.import_max_file_size_mb', 2)` — read in `ImportParseRequest::effectiveMaxFileSizeMb()` (`app/Http/Requests/ImportParseRequest.php:15`) and `SuggestFileImportProfileRequest::rules()` (`app/Http/Requests/SuggestFileImportProfileRequest.php:12`) | Server-only; caps `File::max()` validation on the upload endpoints | `config/yaffa.php:12`, defaults to `2` (MB) if unset | Low. Enforced before the file is even opened by League/CSV or the QIF parser (Laravel's own `File` validation rule rejects oversized uploads). Not client-visible as a config value — the browser has no way to know this cap except by trial and error, so an oversized upload always round-trips to the server before being rejected. |
| `IMPORT_MAX_ROWS` | `config('yaffa.import_max_rows', 5000)` — read in `CsvParserService::parseFile()` (`app/Services/Import/CsvParserService.php:54`) and twice in `QifParserService::doParseLines()` (`app/Services/Import/QifParserService.php:151,221`) | Server-only; caps the number of parsed rows/entries per request, mid-stream (throws `RuntimeException` once exceeded, so the file is read up to the cap, not rejected up front by size alone) | `config/yaffa.php:13`, defaults to `5000` if unset | Low-medium. This is the actual bound on per-request CPU/memory for parsing; it is *not* a hard byte-size cap, so a file with very long lines/columns under `IMPORT_MAX_ROWS` rows can still be large within the `IMPORT_MAX_FILE_SIZE_MB` ceiling. Combined with no rate limiting on the endpoint (see `architecture.md` § Known risks), repeated near-cap uploads are the realistic abuse case, not any single request. |
| `PRISM_REQUEST_TIMEOUT` | `config('prism.request_timeout', 20)` inside the Prism package, consumed transitively by `AiImportProfileSuggestionService::callAiProvider()` (`app/Services/Import/AiImportProfileSuggestionService.php:118`) | Server-only | `config/prism.php:8`, `.env.example:123` | Low, but **mislabeled in `.env.example`**: the comment directly above it (`.env.example:122`, "AI import profile suggestion timeout (seconds)") implies this is specific to this feature. It is actually the global Prism request timeout, shared with AI document processing (OCR/receipt extraction) and any other Prism-backed feature. A future reader tuning this for import-suggestion latency would inadvertently change timeout behavior for unrelated AI calls. |

## Gap: Env Vars Referenced in `SPECIFICATION.md` but Not Actually Undocumented in `.env.example`

`IMPORT_MAX_FILE_SIZE_MB` and `IMPORT_MAX_ROWS` **are implemented** — this contradicts an assumption in this doc's original task brief, which expected them to be spec-only and unimplemented. They are real, working config knobs (see table above). The actual gap is narrower than "unimplemented": **neither variable appears anywhere in `.env.example`** (`grep -n "IMPORT_MAX" .env.example` returns nothing). An operator reading `.env.example` top-to-bottom would never learn these exist or what their defaults are; they'd only be discoverable by reading `config/yaffa.php` or this document. Recommendation for a follow-up (not made here, since `.env.example` edits weren't in scope): add both to `.env.example` near the other `AI_DOCUMENT_*`/`TESSERACT_*` feature-scoped settings.

## Not Applicable to This Feature (checked, ruled out)

- **Per-provider API keys** (`OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, etc. in `config/prism.php:10-70`, sourced from `.env`): these are **global fallback** provider credentials for the Prism package. `AiImportProfileSuggestionService::callAiProvider()` always calls `.usingProviderConfig(['api_key' => $config->api_key])` with the **per-user, DB-stored, `encrypted`-cast** key from `AiProviderConfig` (`app/Models/AiProviderConfig.php:53`), which overrides the env-level default for that call. So none of these global env-level API keys are actually exercised by this feature's code path — they exist for other Prism consumers in the app (e.g. AI document processing) or as a fallback that this feature's calls always bypass.
- No secret/credential introduced by this feature is ever sent to the browser. The AI suggestion response returned to the client (`FileImportProfileApiController::suggest`) contains only the structured profile suggestion (delimiter, mapping, notes) — never the API key, provider config, or any other secret.
